## Project Overview

Media Server Pro 3 is a modular, fault-tolerant media streaming server written in Go. It provides video and audio
streaming with HLS adaptive streaming, thumbnail generation, analytics, user authentication, mature content scanning,
and an admin panel. The frontend is a React 19 + TypeScript + Vite SPA (on the `react` branch) served from
`web/static/react/`.

### HLS Pre-Generation Tool

The HLS pre-generation tool (`cmd/hls-pregenerate/`) allows batch generation of HLS streams before server runtime. It
shares the same cache directory and job state with the server, enabling seamless reuse of generated content.

**Building:**

```bash
# Build the pregenerate tool
go build -o hls-pregenerate.exe ./cmd/hls-pregenerate

# For Linux/Mac
go build -o hls-pregenerate ./cmd/hls-pregenerate
```

**Usage:**

```bash
# Generate HLS for all videos with default qualities (480p, 720p, 1080p)
.\hls-pregenerate.exe

# Generate specific qualities
.\hls-pregenerate.exe -quality "480p,720p"

# Use multiple workers for faster processing
.\hls-pregenerate.exe -workers 4

# Process only specific files
.\hls-pregenerate.exe -filter "*.mp4"

# Dry run (preview without generating)
.\hls-pregenerate.exe -dry-run

# Skip files that already have HLS streams
.\hls-pregenerate.exe -skip-done=true

# Use custom config file
.\hls-pregenerate.exe -config custom-config.json

# Override video directory
.\hls-pregenerate.exe -video-dir "/path/to/videos"
```

**How it works:**

1. The tool uses the **same HLS module** as the server (`internal/hls/`)
2. Generated files are stored in the **same cache directory** (`Directories.HLSCache` from config)
3. Job state is saved to `{cacheDir}/jobs.json` **every 30 seconds** during generation
4. The server **automatically discovers** existing HLS files on startup via disk scanning
5. Both server and tool use **MD5 hash of media path** as job ID, ensuring consistent identification

**Crash Recovery:**

- Job state is saved periodically (every 30s) and after each file completes
- On server startup, the HLS module scans the cache directory and reconstructs job entries for any existing HLS content
- This ensures HLS files generated by the tool are always recognized, even if `jobs.json` is lost or corrupted

**Best Practices:**

- Run the pregenerate tool during off-peak hours or before server deployment
- Use `-workers` flag to control concurrent ffmpeg processes (default: 2)
- Enable `-skip-done` to avoid re-generating existing content
- Monitor disk space - HLS files can be large (roughly 1.5x-2x original file size across all qualities)

### Offline Diagnostic Tool (media-doctor)

The offline diagnostic tool (`cmd/media-doctor/`) checks system health, validates the HLS cache, and repairs issues *
*while the server is stopped**. HLS files are **never deleted unless corrupt** — valid streams are preserved permanently
since generation is expensive.

**Building:**

```bash
go build -o media-doctor.exe ./cmd/media-doctor

# For Linux/Mac
go build -o media-doctor ./cmd/media-doctor
```

**Usage:**

```bash
# Run all checks (read-only, no changes made)
.\media-doctor.exe

# Use a specific config file
.\media-doctor.exe -config config.json

# Also check database connectivity
.\media-doctor.exe -check-db

# Also verify all media files are readable
.\media-doctor.exe -check-media

# Show passing checks too (not just failures)
.\media-doctor.exe -verbose

# Preview what -fix would change without writing anything
.\media-doctor.exe -fix -dry-run

# Apply fixes: remove stale locks, remove corrupt HLS dirs, rebuild jobs.json
.\media-doctor.exe -fix

# Full diagnostic with all checks and fixes applied
.\media-doctor.exe -fix -check-db -check-media -verbose
```

**What it checks:**

1. **System** — ffmpeg/ffprobe in PATH, disk space across all configured directories
2. **Database** (`-check-db`) — MySQL connectivity, schema version, row counts per table
3. **Directories** — All configured paths exist and are writable
4. **Media files** (`-check-media`) — All video/audio files are readable, no zero-byte files
5. **HLS cache** — master.m3u8 integrity, variant playlists, TS segment sync-byte spot-check, stale lock files

**What `-fix` does:**

- Removes `.lock` files left behind when the server was stopped mid-generation (server is offline so all locks are
  stale)
- Removes HLS directories that are **corrupt** (missing master playlist, missing segments, invalid TS data)
- **Never removes valid completed HLS streams**
- Rebuilds `jobs.json` to reflect actual on-disk state (adds untracked dirs, removes orphaned entries)

**HLS permanence policy:**

- Valid HLS streams are kept forever — generation costs too much CPU/time to repeat
- Only corrupt or incomplete (lock-file + invalid files) HLS directories are cleaned
- After `-fix`, the server will immediately see all valid streams on next startup

**Uses same config as server:**
Reads `config.json` and `.env` using the exact same config loading logic as the server.
All directory paths, database credentials, and HLS settings come from your existing config.

## Architecture

### Modular System

The server uses a **modular architecture** where each feature is an independent module. All modules implement the
`server.Module` interface:

```go
type Module interface {
    Name() string
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
    Health() models.HealthStatus
}
```

**Critical modules** (whose failure prevents server startup):

- `database` - MySQL persistence (required for all data storage)
- `security` - Rate limiting, CORS, CSP, IP filtering
- `auth` - User and admin authentication
- `media` - Media library management and metadata
- `streaming` - File streaming with range support
- `tasks` - Background task scheduler
- `scanner` - Mature content detection
- `thumbnails` - Thumbnail generation

**Non-critical modules** (can fail gracefully):

- `hls` - HLS adaptive streaming generation (falls back to direct streaming)
- `analytics` - View tracking and statistics
- `playlist` - Playlist management
- `admin` - Admin panel operations
- `upload` - File upload handling
- `validator` - File validation
- `backup` - Backup/restore functionality
- `autodiscovery` - Auto-discovery of media files
- `suggestions` - Content recommendations
- `categorizer` - Media categorization
- `updater` - Software updates
- `remote` - Remote media source support

### Request Flow

1. **Request arrives** → `api/routes/routes.go` (Gorilla Mux router)
2. **Middleware chain** (from `pkg/middleware/`):
    - Session authentication (`makeSessionAuth`)
    - Rate limiting (security module)
    - CORS (security module)
    - Admin auth check (`makeAdminAuth` for admin routes)
3. **Handler** (from `api/handlers/`) processes request
4. **Module** (from `internal/`) performs business logic
5. **Response** returned to client

### Configuration

Configuration is loaded in this order (later sources override earlier ones):

1. `config.json` - Primary configuration file
2. `.env` - Environment variable overrides (use `.env.example` as template)

Key configuration sections:

- `server` - Host, port, timeouts, HTTPS settings
- `directories` - File paths for videos, music, thumbnails, uploads, etc.
- `database` - MySQL connection settings, pool config (required)
- `streaming` - Chunk sizes, adaptive streaming settings
- `features` - Feature toggles (HLS, analytics, uploads, etc.)
- `security` - Rate limits, CORS, CSP settings
- `auth` - Session timeouts, login attempts, guest access

### Background Tasks

The `tasks` module implements a cron-like scheduler. Tasks are registered in `cmd/server/main.go`:

- **media-scan** (1 hour) - Scans directories for new media
- **metadata-cleanup** (24 hours) - Removes metadata for deleted files
- **thumbnail-generation** (30 minutes) - Generates missing thumbnails
- **mature-content-scan** (12 hours) - Scans for mature content
- **health-check** (5 minutes) - System diagnostics and disk space monitoring

All tasks support context cancellation and graceful shutdown.

### Frontend Integration

- HTML templates are embedded using `//go:embed` in `web/server.go`
- Static assets (CSS/JS) are also embedded and served at `/web/static/`
- Templates are served directly from memory (no disk I/O after build)
- Main pages: index, login, signup, admin-login, profile, player, admin

## Database Architecture

### Overview

The server uses **MySQL database** exclusively for all data persistence. The system uses the **Repository Pattern** for
clean abstraction of data access.

### Database Module (`internal/database/`)

The database module manages MySQL connections, migrations, and health monitoring:

**Files:**

- `database.go` - Core module with connection pooling, retry logic, health checks
- `migrations.go` - Migration framework using embedded SQL files
- `sql/001_initial_schema.sql` - Initial database schema (13 tables)

**Key Features:**

- Connection pooling with configurable limits
- Automatic retry logic with exponential backoff
- Transaction-safe migrations with version tracking
- Health monitoring via `Health()` method
- Critical module - server will not start without database connection

### Repository Pattern (`internal/repositories/`)

Repositories provide a consistent interface for data access, abstracting the underlying storage:

**Structure:**

```
internal/repositories/
  ├── interfaces.go           - Repository contracts
  └── mysql/
      ├── user_repository.go    - MySQL implementation
      └── session_repository.go - MySQL implementation
```

**Interfaces:**

- `UserRepository` - CRUD operations for users
- `SessionRepository` - Session management

**Benefits:**

- Clean separation of business logic and persistence
- Easy testing with mock repositories
- Transaction support for multi-table operations
- Prepared statements for security and performance

### Database Schema

The initial schema (v1) includes 13 tables:

**Core Tables:**

1. `users` - User accounts with indexed username/email
2. `user_permissions` - Granular permission control
3. `user_preferences` - UI and playback preferences
4. `sessions` - Active user sessions with expiry

**Media Tables:**

5. `media_metadata` - File metadata, view counts, mature flags
6. `media_tags` - Tag associations
7. `playback_positions` - Resume positions per user

**Playlist Tables:**

8. `playlists` - User playlists
9. `playlist_items` - Playlist contents

**Analytics/Admin Tables:**

10. `analytics_events` - User activity tracking
11. `audit_log` - Administrative actions
12. `scan_results` - Mature content scan results
13. `scan_reasons` - Flagging reasons

**Schema Features:**

- InnoDB engine with ACID guarantees
- UTF8MB4 character set for full Unicode support
- Foreign keys with CASCADE deletes for referential integrity
- Indexes on frequently queried columns
- Proper NULL handling for optional fields

### How It Works

**Startup Flow:**

1. **Config Loading** - Database config loaded from `config.json` and `.env`
2. **Database Module Start** - Attempts MySQL connection with retry logic
3. **Migration Execution** - Applies pending schema migrations in transactions
4. **Repository Initialization** - Creates MySQL repository instances
5. **Cache Population** - Loads users/sessions into memory for performance

**Runtime Operation:**

- All data access goes through repository interfaces
- MySQL repositories use prepared statements and transactions
- In-memory cache updated after repository operations
- Background cleanup tasks use repository methods

**Critical Module:**

- Database is marked as a critical module
- If MySQL connection fails, the server will not start
- Ensures data consistency and prevents operation without persistence
- Connection is retried according to `DATABASE_MAX_RETRIES` setting

**Module Integration:**
Currently integrated with:

- ✅ **Auth module** - Uses UserRepository and SessionRepository
- ⏳ **Media module** - Can use MediaMetadataRepository (future)
- ⏳ **Analytics module** - Can use AnalyticsRepository (future)
- ⏳ **Playlist module** - Can use PlaylistRepository (future)

### Adding Database Support to Other Modules

**Example: Adding MediaMetadataRepository to Media Module**

1. **Define Interface** (`internal/repositories/interfaces.go`):

```go
type MediaMetadataRepository interface {
    Upsert(ctx context.Context, path string, metadata *models.MediaMetadata) error
    Get(ctx context.Context, path string) (*models.MediaMetadata, error)
    Delete(ctx context.Context, path string) error
    List(ctx context.Context, filter models.MediaFilter) ([]*models.MediaMetadata, error)
}
```

2. **Implement MySQL Repository** (`internal/repositories/mysql/media_repository.go`):

```go
type MediaMetadataRepository struct {
    db  *sql.DB
    log *logger.Logger
}

func NewMediaMetadataRepository(db *sql.DB) repositories.MediaMetadataRepository {
    return &MediaMetadataRepository{db: db, log: logger.New("media-repo")}
}

func (r *MediaMetadataRepository) Upsert(ctx context.Context, path string, metadata *models.MediaMetadata) error {
    _, err := r.db.ExecContext(ctx, `
        INSERT INTO media_metadata (path, views, category, is_mature)
        VALUES (?, ?, ?, ?)
        ON DUPLICATE KEY UPDATE views=?, category=?, is_mature=?
    `, path, metadata.Views, metadata.Category, metadata.IsMature,
       metadata.Views, metadata.Category, metadata.IsMature)
    return err
}
// ... implement other methods
```

3. **Update Module Constructor** (`internal/media/media.go`):

```go
func NewModule(cfg *config.Manager, dbModule *database.Module) (*Module, error) {
    if dbModule == nil || !dbModule.IsConnected() {
        return nil, fmt.Errorf("database module is required")
    }

    metadataRepo := mysql.NewMediaMetadataRepository(dbModule.DB())

    return &Module{
        metadataRepo: metadataRepo,
        // ... other fields
    }, nil
}
```

4. **Update main.go** to pass database module and handle error from constructor

### Best Practices

**Repository Implementation:**

- Use transactions for multi-table operations
- Use prepared statements to prevent SQL injection
- Handle `sql.ErrNoRows` gracefully
- Use context for timeout/cancellation
- Log errors with sufficient context

**Migration Files:**

- Name with version prefix: `001_description.sql`, `002_description.sql`
- Include `ENGINE=InnoDB` and `CHARSET=utf8mb4`
- Add indexes for frequently queried columns
- Use foreign keys for referential integrity
- Never modify existing migrations (create new ones)

**Testing:**

- Use repository interfaces for easy mocking
- Test MySQL implementations thoroughly
- Test transaction rollback scenarios
- Test connection failure and retry logic

### Troubleshooting

**Connection Issues:**

- Check MySQL is running: `systemctl status mysql` (Linux) or Services panel (Windows)
- Verify credentials: `mysql -u mediaserver -p mediaserver`
- Check firewall: `netstat -an | grep 3306`
- Review logs: `[database]` prefix in server logs

**Migration Errors:**

- Check schema_migrations table: `SELECT * FROM schema_migrations;`
- Verify SQL syntax in migration files
- Ensure database user has ALTER/CREATE permissions
- Check for existing conflicting tables

**Performance:**

- Monitor connection pool: Check `DATABASE_MAX_OPEN_CONNS`
- Enable slow query log in MySQL
- Add indexes for slow queries
- Consider increasing `DATABASE_TIMEOUT` for complex queries

## Common Patterns

### Adding a New Module

1. Create package in `internal/mymodule/`
2. Implement `server.Module` interface (Name, Start, Stop, Health)
3. Add constructor `NewModule(cfg *config.Manager) *Module`
4. Register in `cmd/server/main.go`:
    - Create instance
    - Add to modules slice
    - Add to criticalModules map if critical
5. Inject into handlers via `HandlerDeps` if needed

### Adding a New API Endpoint

1. Add handler method in `api/handlers/handlers.go`
2. Register route in `api/routes/routes.go` under appropriate section:
    - Public routes (no auth required)
    - User routes (requires session auth)
    - Admin routes (requires admin auth)
3. Apply middleware as needed (`makeSessionAuth`, `makeAdminAuth`)

### Adding a Background Task

Register in `cmd/server/main.go` → `registerTasks()`:

```go
scheduler.RegisterTask(
    "task-id",
    "Task Name",
    "Description",
    interval,  // e.g., 30*time.Minute
    func(ctx context.Context) error {
        // Task implementation
        // Check ctx.Done() for cancellation
        return nil
    },
)
```

### Module Dependencies

Modules receive dependencies through their handlers via `HandlerDeps`. The handler accesses modules via:

```go
h.media.ListMedia(filter)
h.streaming.StreamFile(path)
h.auth.ValidateSession(sessionID)
```

Do not create circular dependencies between modules. If two modules need to communicate, pass data through handlers or
use shared models in `pkg/models/`.

## Configuration Tips

### Development Setup

1. Copy `.env.example` to `.env`
2. Set `SERVER_PORT=8080` (or higher) to avoid permission issues
3. Set `LOG_LEVEL=debug` for detailed logging
4. Configure media directories to point to your test media

### Common Issues

- **Port binding failed**: Change `SERVER_PORT` in `.env` or `config.json`
- **Module startup failed**: Check logs for missing directories or permissions
- **Thumbnails not generating**: Ensure ffmpeg is installed and in PATH
- **HLS streaming not working**: Check `HLS_ENABLED=true` and ffmpeg availability
- **Database connection failed**: Check MySQL is running, verify credentials, ensure `DATABASE_ENABLED=true`
- **Build fails with "undefined: getDiskUsage"**: Use `go build ./cmd/server` instead of `go build cmd/server/main.go`
