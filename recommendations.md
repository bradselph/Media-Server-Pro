**Scope:** Features, infrastructure, and best practices the project is missing

This is not a bug list — see `audit-report-2026-03-14.md` for that. These are gaps that affect production readiness, developer experience, and user experience.

## Infrastructure & DevOps

### P0: Dockerfile + docker-compose.yml
No container support exists. Contributors must install Go, Node, ffmpeg, and MySQL manually. A multi-stage Dockerfile (build + runtime) and a compose stack (server + MySQL + optional slave) would make onboarding trivial and enable container-based deploys.

### P0: README.md
No README at the project root. `CLAUDE.md` is AI guidance, not user-facing documentation. Need: project overview, quick start, configuration reference, architecture diagram, screenshots.

### P0: .env.example + config.json.example
Neither exists. `.env` is only generated by the interactive `setup.sh` wizard. A committed template with documented variables lets developers and CI configure without running the wizard.

### P1: Activate git hooks
`scripts/pre-push-check.sh --install-hook` exists but has never been run — `.git/hooks/` has only `.sample` files. Broken code can be pushed without any local checks.

### P2: CodeQL / SAST workflow
CI runs `govulncheck` + `npm audit` but has no static application security testing producing GitHub Security tab results. `.gitignore` references `*.sarif` — suggesting this was considered but never wired up.

## Testing

### P0: Go integration tests
Zero integration tests exist. All 11 test files are pure unit tests. No test verifies that an HTTP route responds correctly end-to-end, or that a repository correctly reads/writes to a database.

**Recommended starting points:**
- Auth flow: login -> session -> protected endpoint -> logout
- Media list + filter + pagination
- Playlist CRUD
- Receiver catalog push + proxy

### P0: Repository layer tests
All 22 GORM repositories in `internal/repositories/mysql/` are completely untested. Use testcontainers-go with a MySQL container, or an in-memory SQLite for fast unit tests.

### P1: Race detector in CI
`go test` runs without `-race`. The codebase has extensive concurrent map access protected by `sync.RWMutex`. Adding `-race` to CI would catch any missed mutex guard automatically.

### P1: Code coverage in CI
`go test` doesn't use `-coverprofile`. No coverage badge, threshold enforcement, or Codecov upload. The frontend has `vitest --coverage` configured but only 1 test file to measure.

### P1: Frontend component tests
Only 1 test file (`client.test.ts`) exists for the entire React frontend (6 pages, 19 admin tabs, 7 components, 5 stores, 4 hooks). Vitest + Testing Library are already installed — they just need tests written.

### P2: Go benchmarks
No `Benchmark*` functions anywhere. Critical paths (HLS segment serving, concurrent stream handling, media list filtering at 50K items) have no performance baseline.

### P2: Go fuzz targets
No `Fuzz*` functions. Range header parsing, path resolution, and JSON parsing all handle untrusted input and would benefit from fuzzing.

### P2: E2E tests
No Playwright or Cypress. At minimum, a smoke test covering login -> browse -> play -> logout would catch integration regressions.

### P2: Load testing scripts
No k6, Locust, or wrk scripts. For a media server designed to serve concurrent HLS streams, there's no way to characterize throughput limits.

## Observability

### P1: Structured log fields
Even in JSON mode, logs are free-form `%v` text. No discrete typed fields (`user_id`, `media_id`, `duration_ms`, `bytes_sent`). Log aggregators can't reliably parse these without regex. Add structured fields to the logger.

### P2: Disk space metrics
The health-check task checks directory existence but doesn't measure free space. Media servers fill disks — this needs monitoring.

## API & Backend

### P1: Machine-readable error codes
API responses only carry `{"success": false, "error": "plain text"}`. No `code` field (e.g., `"MEDIA_NOT_FOUND"`, `"RATE_LIMIT_EXCEEDED"`) for clients to branch on programmatically. Consider RFC 7807 Problem Details.

### P1: Rate limit headers
Per-IP rate limiting exists but 429 responses include no `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, or `Retry-After` headers. Clients can't adapt.

### P2: Circuit breakers
No circuit breaker on any external dependency (MySQL, HuggingFace API, remote media sources). If MySQL goes slow (not down), all goroutines pile up. Use a circuit breaker pattern for DB calls and external APIs.

### P2: WebSocket message size limits
`conn.SetReadLimit()` is never called on the receiver WebSocket. A compromised slave could send an arbitrarily large JSON message and exhaust master memory.

### P2: Slave reconnection with backoff
If the master restarts, the slave must be manually restarted. No exponential backoff reconnection logic exists in `cmd/media-receiver/`.

### P2: Consistent pagination envelope
Some endpoints return `{total_items, total_pages, items}`, others return bare arrays. Standardize on a uniform shape with cursor-based pagination for high-cardinality endpoints.

### P2: Full-text search
Search is a linear O(n) in-memory substring match on `item.Name`. No MySQL `FULLTEXT` index, no fuzzy matching, no relevance ranking. At 50K+ items this will be slow and imprecise.

### P2: Config hot-reload
The `OnChange` callback mechanism exists in the config manager but is never wired to any module. Config changes require a restart.

## Frontend — Player & Playback

### P1: MediaSession API
No `navigator.mediaSession` integration. OS-level media controls (lock screen, notification shade, media keys on Bluetooth headphones) show nothing — no title, no artwork, no play/pause/seek controls.

### P1: PiP event listeners
`requestPictureInPicture()` works but there are no `enterpictureinpicture`/`leavepictureinpicture` event listeners. Closing PiP via the browser's native button leaves the player state inconsistent.

### P2: Chromecast support
No Google Cast SDK integration. For a media server, casting to a TV is a core feature.

### P2: Mobile gestures
No swipe-to-seek, double-tap-to-seek, swipe-down-to-dismiss, or pinch-to-zoom on the video element. Only touch-drag on the progress bar exists.

### P2: Autoplay policy handling
`el.play()` called unconditionally. Browser autoplay policies may block silently on mobile. Need a "tap to play" overlay when autoplay is blocked.

## Frontend — UX & Polish

### P0: Accessibility (focus management)
No focus traps on modals/panels. Admin tabs use `<button>` without `role="tab"`/`aria-selected`. No skip-to-content link. Keyboard focus rings are minimal — icon buttons have none.

### P1: Dynamic page titles
`document.title` is never updated. Every page shows "Media Server Pro" regardless of context. Playing a video should show the media name in the browser tab. Use `react-helmet` or a route-level `useEffect`.

### P1: Open Graph meta tags
No `og:title`, `og:image`, `og:description`. Sharing a player URL in messaging apps produces no preview card.

### P1: Admin panel responsive CSS
`admin.css` has zero `@media` queries. On a phone screen, the multi-column stats grid and tab navigation overflow. The admin panel is unusable on mobile.

### P1: Password change in profile
The profile page shows account info but has no way to change the password from the UI.

### P1: User session management
Users cannot see or revoke their own active sessions from the profile page. Only admins can manage sessions.

### P2: Dark mode runtime OS sync
`theme='auto'` resolves `prefers-color-scheme` once at load time but doesn't listen for system theme changes. If the OS switches light/dark while the app is open, the UI doesn't update.

### P2: Skeleton screens
Player page returns `null` while metadata loads — no skeleton. Admin tabs show nothing during first fetch. No global loading indicator (NProgress-style top bar) during navigation.

### P2: Tag browsing UI
Tags exist in the data model and are filterable via URL params, but there's no tag cloud, tag page, or clickable tag chips on media cards.

### P2: Favorites / bookmarks
No way for a user to mark media as a favorite and filter by favorites. Distinct from playlists (which are sequential playback queues).

### P2: Drag-and-drop playlist reorder
`playlistStore.reorderItems(fromIndex, toIndex)` exists but there's no drag UI. Playlists display in order with no way to reorder via mouse or touch.

### P2: Analytics time-series charts
`DailyStats` type exists, `adminApi.getDailyStats` endpoint exists, but `AnalyticsTab` never calls it. No date-range chart is rendered. All "charts" are raw CSS div bars — no chart library.

## Frontend — Performance

### P1: Admin page code splitting
All 19 admin tab components load when `/admin` is first accessed. Each tab should be individually `React.lazy`-loaded since most admins only visit 2-3 tabs per session.

### P2: Responsive images
Thumbnails served as-is (JPEG/PNG). No `<picture>` element with `srcset` for WebP/AVIF with fallback. No `sizes` attribute for responsive image sizing. WebP is generated server-side but the frontend doesn't negotiate formats.

### P2: Prefetch next playlist item
No prefetching of the next item in a playlist. Starting playback of the next track requires a full metadata + stream request.
